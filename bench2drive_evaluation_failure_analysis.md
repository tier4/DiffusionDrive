## Executive Summary

The Bench2Drive adaptation of DiffusionDrive fails at open-loop evaluation for **two independent categories of reasons**:

1. **The evaluation pipeline does not exist for Bench2Drive.** The `run_pdm_score.py` script is hardwired to NAVSIM data formats, nuPlan map APIs, and pre-computed MetricCache files. None of these have Bench2Drive equivalents. The evaluation literally cannot load B2D data.

2. **Even if evaluation worked, the training pipeline has critical spatial mismatches** that would produce a broken model. The B2D data uses 85m BEV range but the model internally assumes 64m, creating a 1.33x scale error in every spatial operation.

Additionally, **Bench2Drive is fundamentally a closed-loop benchmark** (live CARLA execution). The PDM-Score open-loop evaluation used by NAVSIM is not the correct evaluation approach for B2D. The Bench2DriveZoo already has working open-loop L2 evaluation code that should be used instead.

## Category 1: Evaluation Pipeline Is Missing

### Issue 1.1: No MetricCache Generation for Bench2Drive (BLOCKER)

**What it is:** `run_pdm_score.py` requires pre-computed `MetricCache` files containing ground-truth ego states, expert trajectories, observations, centerlines, route lane IDs, and drivable area maps. These are generated by `run_metric_caching.py`.

**Why it fails:** The metric caching pipeline (`metric_cache_processor.py`) is entirely dependent on nuPlan infrastructure:
- Line 54 of `caching.py`: wraps every scene into `NavSimScenario(scene, map_root=os.environ["NUPLAN_MAPS_ROOT"])`
- `NavSimScenario.map_api` requires nuPlan map files (`us-nv-las-vegas-strip` etc.)
- `PDMClosedPlanner` queries `map_api.get_proximal_map_objects()` for drivable areas
- `PDMDrivableMap.from_simulation()` uses `SemanticMapLayer.ROADBLOCK`, `SemanticMapLayer.INTERSECTION`

None of this exists for CARLA/Bench2Drive maps. No adapter was written. **Without MetricCache, there is nothing to score against.**

### Issue 1.2: SceneLoader Cannot Load Bench2Drive Data (BLOCKER)

**What it is:** `run_pdm_score.py` (lines 60-65, 115-120) exclusively uses `SceneLoader` from `navsim.common.dataloader`.

**Why it fails:** `SceneLoader` reads NAVSIM pickle logs from `OPENSCENE_DATA_ROOT`. Bench2Drive uses `Bench2DriveSceneLoader` which reads `.json.gz` annotation files from a completely different directory structure. These two classes share no common interface. The evaluation script cannot even enumerate B2D scenes.

### Issue 1.3: nuPlan Map API Has No CARLA Adapter (BLOCKER)

**What it is:** The PDM scorer needs drivable area maps, centerlines, and route lane IDs from nuPlan's `AbstractMap` interface.

**Why it fails:** Bench2Drive uses CARLA HD maps stored as `.npz` files with lane markings and trigger volumes. Nobody wrote a `CARLAMapAdapter` implementing nuPlan's map interface. The BEV generation code (`bev_map_utils.py`) rasterizes CARLA maps directly, bypassing the nuPlan API entirely, but the evaluation pipeline doesn't use BEV rasterization for scoring.

### Issue 1.4: Coordinate System Mismatch in Scoring

**What it is:** `pdm_score.py` transforms predicted trajectories from ego frame to global frame using `relative_to_absolute_poses()` from nuPlan, which assumes nuPlan's right-handed coordinate system.

**Why it fails:** Bench2Drive data is in CARLA's left-handed coordinate system (Y-right, yaw-clockwise). The B2D pipeline explicitly does NOT convert to right-handed (`transform_points_to_ego(left_to_right=False)`, and the right-handed path raises `NotImplementedError`). All spatial metrics (collision, drivable area, progress) would be computed in the wrong coordinate frame.

### Issue 1.5: Bench2Drive Is Not an Open-Loop Benchmark

**What it is:** PDM-Score is designed for open-loop trajectory evaluation (predict trajectory, compare to expert, simulate with bicycle model).

**Why it doesn't apply:** Bench2Drive is explicitly a **closed-loop benchmark**. The paper states this clearly, and the Bench2Drive README notes: *"L2 error is not a meaningful indicator at all."* The official metrics (Driving Score, Success Rate, Multi-Ability) come from running agents live in CARLA 0.9.15 across 220 pre-defined routes.

**What exists instead:**
- **Closed-loop:** `/home/chenglin/workspace/Bench2DriveZoo/team_code/diffusiondrive_b2d_agent.py` wraps the model for CARLA execution
- **Open-loop L2:** `/home/chenglin/workspace/Bench2DriveZoo/adzoo/diffusiondrive/test_openloop_vad.py` computes VAD-compatible L2 metrics (Planning_L2_05s through Planning_L2_40s) directly from B2D data, without needing MetricCache or nuPlan maps

## Category 2: Training Pipeline Spatial Mismatches

Even if evaluation is fixed, the trained model will produce poor results due to these issues.

### Issue 2.1: LiDAR BEV 85m-vs-64m Scale Mismatch (CRITICAL)

**Where:** `transfuser_features_b2d.py:257` creates a 340x340 histogram at 85m range, resizes to 256x256.

**The problem:** The model's `GridSampleCrossBEVAttention` (`modules/blocks.py:83-86`) normalizes trajectory points using `config.lidar_max_x = 32` and `config.lidar_max_y = 32` (inherited from `TransfuserConfig`, never overridden by `Bench2DriveConfig`). This means:

- Model thinks: 256 pixels = 64m, so 1 pixel = 0.25m
- Reality: 256 pixels = 85m, so 1 pixel = 0.332m
- **Scale error: 1.328x**

A real object at 32m is at pixel ~96 in the B2D BEV, but the model interprets pixel 96 as being at 24m. Every cross-attention spatial query is wrong.

### Issue 2.2: BEV Semantic Map Resolution Mismatch (CRITICAL)

**Where:** `bench2drive_constants.py` defines BEV at 0.332 m/pixel. `transfuser_config.py` defines `bev_pixel_size = 0.25`.

**The problem:** Same 1.33x scale error as the LiDAR BEV, but now affecting:
- BEV semantic loss (supervision targets at wrong scale)
- The BEV semantic head's upsampling logic
- Any module that converts between BEV pixel coordinates and metric coordinates

### Issue 2.3: Agent Detection Range Overflow (HIGH)

**Where:** `bench2drive_scene.py:732` uses 42.5m circular filtering. `transfuser_model_v2.py:174-175` clamps agent positions with `tanh() * 32`.

**The problem:** Agents in the 32m-42.5m range are included in training supervision but are unrepresentable by the model's output head. These create irreducible loss and bias the detector toward underestimating distant objects.

### Issue 2.4: Config Fields Never Overridden (HIGH)

**Where:** `bench2drive_config.py` inherits `TransfuserConfig` and only adds normalization parameters.

**The problem:** These NAVSIM-specific spatial parameters are silently inherited and never updated for B2D:

| Parameter | Inherited Value | Should Be (for 85m B2D) |
|-----------|----------------|------------------------|
| `lidar_min_x` | -32 | -42.5 |
| `lidar_max_x` | 32 | 42.5 |
| `lidar_min_y` | -32 | -42.5 |
| `lidar_max_y` | 32 | 42.5 |
| `bev_pixel_size` | 0.25 | 0.332 |

Every internal module that reads these config values operates on the wrong geometry.

### Issue 2.5: Camera Stitching Geometry Mismatch (MODERATE)

**Where:** `transfuser_features_b2d.py:161-179` vs `transfuser_features.py:62-75`.

**The problem:** Original NAVSIM crops side cameras `[28:-28, 416:-416]` before stitching, limiting the lateral FOV and creating a specific perspective geometry. B2D uses full 900x1600 images without any cropping. The stitched image has different FOV, horizon placement, and object scale. Pretrained ResNet-34 visual features (initialized from `bkb_path`) will not transfer well.

### Issue 2.6: Velocity Sign Convention Issue (MODERATE)

**Where:** `bench2drive_scene.py:553-558`:
```python
vy = -speed * sin(theta)  # "Negative for CARLA CW rotation"
```

**The problem:** In CARLA's left-handed coordinate system (+X forward, +Y right), a heading of +90° (pointing along +Y) should give `vy = +speed`. The negation produces `-speed`, which is inconsistent with CARLA's coordinate conventions. The ego velocity fed to the status encoding has incorrect Y-component for non-zero heading angles.

### Issue 2.7: LiDAR np.flipud Axis Fork (MODERATE)

**Where:** `transfuser_features_b2d.py:249` applies `np.flipud(hist)`.

**The problem:** The original NAVSIM LiDAR builder does not flip. This was likely added empirically to fix visualization, but it means the LiDAR Y-axis convention is forked from the original pipeline. Combined with the scale mismatch, it's difficult to verify that LiDAR features, BEV semantic maps, agent positions, and trajectory points are all spatially consistent.

### Issue 2.8: CLAUDE.md Claims Coordinate Conversion That Doesn't Exist (MODERATE)

**Where:** CLAUDE.md line 107: *"CARLA uses a left-handed coordinate system; BEV generation code handles the conversion to right-handed."*

**Reality:** `bev_map_utils.py:207-210`:
```python
if left_to_right:
    raise NotImplementedError("The right-handed conversion is not implemented yet.")
```

The entire B2D pipeline operates in CARLA's left-handed system. NAVSIM's original training data uses nuPlan's right-handed system. When loading pretrained weights, the model's learned spatial features are mirror-flipped relative to B2D inputs.

## Category 3: Architectural Design Issues

### Issue 3.1: Monkey-Patched Normalization Is Fragile

**Where:** `transfuser_model_wrapper.py` replaces `_trajectory_head.norm_odo` and `denorm_odo` at runtime by reassigning instance methods.

**Risk:** If the trajectory head is `deepcopy`'d (common in DDP, EMA, or checkpoint saving), the monkey-patched methods won't survive. This could silently revert to NAVSIM normalization during training or inference.

### Issue 3.2: `dataset_type` Defaults to `"navsim"` (SILENT BUG)

**Where:** `bench2drive_config.py:15`: `dataset_type: str = "navsim"`

**Risk:** If any code path instantiates `Bench2DriveConfig` without explicitly setting `dataset_type: "bench2drive"` (e.g., direct Python construction, test code, or a misconfigured YAML), the normalization silently uses NAVSIM parameters for B2D data. The YAML config does set it correctly, but there's no assertion or validation.

### Issue 3.3: Two Unrelated `Bench2DriveConfig` Classes

**Where:** 
- `navsim/common/bench2drive_dataloader.py` defines `Bench2DriveConfig` (data loading config)
- `navsim/agents/diffusiondrive/bench2drive_config.py` defines `Bench2DriveConfig` (model config)

**Risk:** Import confusion. Both are valid imports with the same class name but completely different semantics.

## Category 4: BEV Generation Pipeline Bugs

The Bench2Drive dataset does NOT come with BEV semantic maps. The team built a custom BEV generation pipeline from scratch using CARLA HD maps (`.npz` files with lane markings and trigger volumes). This pipeline has several issues.

### Issue 4.1: Sample BEV Shows Only 1.6% Road Coverage (CRITICAL)

**Evidence:** Inspecting the test sample BEV cache at `tests/test_data/bench2drive_sample/sample_bev_cache/`:

```
BEV overview (downsampled 8x, # = road, C = centerline):
.............#CC#...............
.............#CC#...............
.............#CC#...............
(repeats for all 32 rows)
```

Road pixels: 786 out of 65,536 total = **1.6%**. The "road" is a 3-4 pixel wide strip (about 1 meter) running vertically through the center. This is a lane marking line, not a filled road surface.

**Root cause:** The sample BEV was generated with the **vector** generator (default `--generation-type vector`), which draws lane boundary lines as thin polylines. The segmentation generator would fill the road surface area. But even if the user ran segmentation mode, the `_generate_from_processor` path calls `generate_bev_from_map()` (the vector function) and then dilates the result, which produces a different road shape than actually filling between lane boundaries.

**Impact:** The model's BEV semantic loss (weight = 14.0!) is training against a map where 98.4% of pixels are background and only thin lane lines exist. In NAVSIM, road polygons fill 20-40% of the BEV. This means:
- The BEV semantic head learns to predict "everything is background"
- The `GridSampleCrossBEVAttention` module gets almost no useful BEV features to cross-attend with
- Spatial reasoning about drivable area is impossible

### Issue 4.2: Resolution Mismatch, 0.332 m/pixel vs 0.25 m/pixel (CRITICAL, restated)

Already covered in Category 2, but directly affects BEV generation. The BEV is generated at `BEV_SEMANTIC_RESOLUTION = 85.0 / 256 = 0.332 m/pixel`. The model config says `bev_pixel_size = 0.25`. So:

- Model assumes 128 rows * 0.25 = 32m forward coverage
- BEV actually covers 128 rows * 0.332 = 42.5m forward
- A vehicle at 20m ahead is at row 67 in B2D BEV, but model interprets that as 15m

This affects the BEV loss, the BEV cross-attention, and any spatial reasoning the model does.

### Issue 4.3: Ego Position Off-by-One After Front Half Extraction (MODERATE)

**Where:** `bev_map_utils.py:44-46` and `bev_semantic_utils.py:91-98`

In the full 256x256 BEV, ego is at row 128 (`height // 2`). After `extract_front_half_bev()` crops `full_bev[:128, :]`, the ego position (row 128) falls **outside** the cropped image (rows 0-127). The dynamic agent overlay in `ego_to_bev_coordinates()` places ego at row 127 (`height - 1`). This creates a **1 pixel (0.332m) systematic offset** between the static road layer and the dynamic vehicle/pedestrian layer.

### Issue 4.4: Default Resolution 0.25 in Agent Overlay Functions (MODERATE)

**Where:** `bev_semantic_utils.py:59` (`ego_to_bev_coordinates(resolution=0.25)`) and line 164 (`generate_agent_mask(resolution=0.25)`)

These functions default to NAVSIM's 0.25 m/pixel resolution instead of B2D's 0.332. The main code path in `bench2drive_scene.py:1030` passes `resolution=BEV_SEMANTIC_RESOLUTION` correctly, so this doesn't trigger in the normal flow. But any direct caller using defaults gets spatially misplaced agents. A maintenance trap.

### Issue 4.5: BEV Orientation Differences vs NAVSIM (HIGH)

**How NAVSIM generates BEV:** The original pipeline in `transfuser_features.py:197-288` uses nuPlan's map API with Shapely geometries. It:
1. Creates the canvas with shape `bev_semantic_frame[::-1]` (swapped H/W)
2. Draws polygons and linestrings with OpenCV
3. Applies `np.rot90(mask)[::-1]` to every layer to fix orientation

This `np.rot90 + vertical flip` corrects from OpenCV's drawing convention to the model's expected BEV orientation (forward = up, right = right).

**How B2D generates BEV:** The pipeline in `bev_map_utils.py` takes a different approach:
1. Creates canvas at normal `(H, W)` shape
2. Manually swaps `[row, col]` to `[col, row]` for OpenCV: `cv2_points = lane_points[:, [1, 0]]`
3. No `np.rot90` or `[::-1]` applied

These two approaches should produce the same result IF the coordinate conversions are perfectly consistent. But the NAVSIM approach uses `_geometry_local_coords` (Shapely affine_transform with nuPlan's right-handed rotation) and `_coords_to_pixel` (places ego at `(0, W/2)` before rotation), while B2D uses `transform_points_to_ego` (CARLA left-handed rotation) and `ego_to_bev_pixels` (places ego at `(H/2, W/2)` directly).

**The risk:** If the B2D BEV has a different orientation convention than what the model's `GridSampleCrossBEVAttention` expects (which was trained with NAVSIM BEVs), the spatial cross-attention will look in the wrong direction. This has not been verified empirically.

### Issue 4.6: Only Classes 0, 1, 3 Present, Missing Classes 2 and 4 (MODERATE)

**Evidence:** All sample BEV files show only classes `{0, 1, 3}` (background, road, centerline). Missing:
- Class 2 (walkway/sidewalk): Not generated from CARLA HD maps. The map data only has lane markings and trigger volumes, not sidewalk polygons.
- Class 4 (static objects): Trigger volumes exist in the map data but are rare and not appearing in the test sample.

In NAVSIM, all 7 classes (0-6) appear in typical BEV maps because the nuPlan map API provides road polygons, walkway polygons, lane centerlines, and static objects directly. The B2D BEV pipeline only has access to lane boundary lines and trigger volumes from the HD map, so the semantic content is fundamentally poorer.

**Impact:** The BEV semantic head has 7 output classes but only ever sees 3 in B2D training data. Classes 2, 4 never appear in the static map (classes 5, 6 come from the dynamic overlay). The model's walkway and static object detection channels are trained on all-zeros, degrading the BEV feature quality for cross-attention.

### Issue 4.7: BEV Cache Saved as float32 Instead of uint8 (MINOR)

**Where:** `generate_bev_cache.py:128-129` saves as `float32`.

The BEV contains integer class labels 0-6. Saving as float32 wastes 4x disk space. Should be uint8.

## Recommended Next Steps (Priority Order)

1. **Stop trying to use `run_pdm_score.py` for B2D.** It cannot work without months of adapter development, and it's the wrong metric anyway.

2. **Use the Zoo's existing open-loop L2 evaluation** (`test_openloop_vad.py`) for development iteration. This gives fast, offline trajectory quality feedback.

3. **Regenerate BEV cache with segmentation mode AND verify road coverage:**
   ```bash
   python3 scripts/generate_bev_cache.py \
       --data-root /path/to/Bench2Drive-Base \
       --map-dir /path/to/Bench2Drive-Map \
       --output-dir "${NAVSIM_EXP_ROOT}/bench2drive_bev_cache_seg" \
       --generation-type segmentation \
       --lane-width 5.0
   ```
   After generation, verify that road pixels cover 15-40% of the BEV (not 1.6%). The current test sample shows vector-mode output where roads are thin lines.

4. **Fix the 85m vs 64m spatial mismatch** before training again. The simplest approach:
   - (a) Crop B2D LiDAR/BEV to 64m to match the model's assumptions (simpler, preserves model architecture), or
   - (b) Override all spatial config fields in `Bench2DriveConfig` to 85m (more work, keeps full range, but requires verifying `GridSampleCrossBEVAttention`, agent head clamp, BEV upsampling, etc.)

5. **Fix the agent filtering radius** to match the model's output range (cap at 32m, or extend the agent head's `tanh * 32` to `tanh * 42.5`).

6. **Verify BEV orientation** empirically: overlay a predicted trajectory on the BEV map and confirm that "forward in trajectory" = "up in BEV". The B2D pipeline uses a fundamentally different rendering approach than NAVSIM (`manual row/col swap` vs `np.rot90[::-1]`).

7. **Verify coordinate/velocity conventions** against actual CARLA ground truth data. Especially the `vy = -speed * sin(theta)` sign.

8. **Target closed-loop CARLA evaluation** as the final benchmark metric.
